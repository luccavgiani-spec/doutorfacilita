# ─────────────────────────────────────────────────────────────
# 4. FLUXO COMPLETO PACIENTE
# ─────────────────────────────────────────────────────────────

  1. Landing (/) → CTA "Consultar agora" leva pra /login
  2. /login → "Cadastre-se" → /cadastrar
  3. /cadastrar (wizard 4 etapas):
       Etapa 1: Dados Pessoais (nome, sobrenome, email, celular*, data nasc, gênero)
       Etapa 2: Endereço (CEP autofill via viacep.com.br, logradouro/número/etc)
       Etapa 3: Documentos (CPF com dígito verificador + alergias chips)
       Etapa 4: Segurança (senha + checklist + contato emergência + aceite termos)
     → supabase.auth.signUp() com metadata completo
     → trigger handle_new_user materializa public.patients + user_roles role='patient'
  4. /login/redirect (server) → detecta role:
       doctor   → /cockpit
       patient  → hasConsultaPaga ? /fila : /checkout
  5. /checkout → tabs PIX/Cartão (UI completa, valores compatíveis com MP)
     → server action finalizarCompra(payload):
         - cria consultation status='created'
         - chama createPayment() (stub: gera STUB-PIX-… ou STUB-CARD-…)
         - marca status='in_queue', paid_at, queued_at, payment_id
         - redirect /fila?consultation=<id>
     → feature flag 'checkout_stub_payment' = true (DEV); desligar em prod
  6. /fila (FilaShell client):
       - JoinConsultButton: estado 'waiting' → escuta postgres_changes em
         consultations filtrado por id; quando doctor_called_at vira não-null,
         estado vira 'ready'
       - Label muda pra "Entrar na Consulta (30s)" com countdown
       - Auto-redirect em 30s (ou clique antes) → ConsultaLive renderiza
         LiveKitRoom fullscreen (video + audio publicados automaticamente)
  7. LiveKit conecta na sala criada pelo create_enter_doc; quando médico
     encerra (end_consultation deleta a sala), paciente recebe Disconnected
     → router.push("/posconsulta")
  8. /posconsulta: card WhatsApp com número mascarado + resumo 4 etapas +
     coluna lateral com 4 cards upsell (Pacote família, Exames, Reconsulta,
     Indique e ganhe). CTAs apontam pra /fila por enquanto (rotas futuras)

# ─────────────────────────────────────────────────────────────
# 5. FLUXO COMPLETO MÉDICO (cockpit)
# ─────────────────────────────────────────────────────────────

  1. /cockpit (CockpitScreen client component):
       Coluna esquerda  : CockpitFila (lista in_queue em tempo real via realtime)
       Coluna central   : visor LiveKit embedded + Mevo card
       Coluna direita   : ChartPanel (3 abas: Prontuário / Histórico / Anamnese)
  2. CockpitFila: SELECT consultations onde status='in_queue' AND doctor_id IS NULL
       - Subscribe postgres_changes em consultations (event='*') → refetch
       - Realtime funciona porque consultations tem REPLICA IDENTITY FULL (mig.034)
       - Tick a cada 30s pra atualizar "Aguardando há Xmin"
       - Card #1 da fila ganha botão "Chamar próximo"
  3. CallNextButton → invoca edge function create_enter_doc:
       - Marca consultation: doctor_id=self, doctor_called_at=now, status=in_progress,
         livekit_room_name='consultation_<id>'
       - Gera AccessToken do doctor (TTL 2h, canPublish/canSubscribe)
       - Devolve {room_name, token, livekit_url}
     → CockpitScreen seta activeCall e renderiza DoctorCallEmbedded no
        doc-video-strip (LiveKitRoom + GridLayout + ParticipantTile + ControlBar minimal)
  4. ChartPanel:
       Aba Prontuário: SOAP (chief_complaint, history_present_illness,
         physical_exam, diagnostic_hypothesis, cid10_codes, conduct).
         Autosave debounced 800ms via upsert em medical_records.
       Aba Histórico: SELECT consultations do mesmo paciente (cross-doctor
         via RLS mig.035). Cada card abre drawer com medical_record completo +
         lista de prescricoes_mevo + signed URLs dos PDFs em storage.
       Aba Anamnese: edita campos persistentes em patients (alergias,
         medical_history, family_history, current_medications). Autosave 800ms.
     ChartPanel expõe flushPending() via useImperativeHandle.
  5. Médico clica "Leave" no DoctorCallEmbedded:
       - CockpitScreen.handleDoctorEndCall:
           a) await chartRef.current?.flushPending() — salva edições em voo
           b) setActiveCall(null) — UI volta ao estado vazio
           c) invoke end_consultation em paralelo
       - end_consultation: RoomServiceClient.deleteRoom (kicka paciente),
         UPDATE consultations SET status='completed', ended_at, duration_seconds
  6. Paciente recebe Disconnected do LiveKit → vai pra /posconsulta

# 15. DECISÕES ARQUITETURAIS CHAVE (registradas)
# ─────────────────────────────────────────────────────────────

- Senha forte gerada pelo admin no cadastro de médico (vs invite por email)
- Modal in-page pra cadastro (vs página dedicada)
- Toggle direto + URL via modal separado nas integrações (vs form único)
- Flush imediato do ChartPanel no encerramento (via useImperativeHandle)
  → garante persistência de qualquer edição em voo no debounce de 800ms
- ChartPanel: prontuário POR CONSULTA (medical_records),
              anamnese POR PACIENTE (patients) — campos persistem entre consultas
- Variáveis de template via sintaxe {{var}} (substituição no cockpit, ainda não impl.)
- has_role usa user_roles (NÃO doctors) — mas RLS de fila usa current_doctor_id
  IS NOT NULL pra evitar dependência circular
- REPLICA IDENTITY FULL em consultations pra realtime entregar UPDATE com RLS
- Topic único por mount no Realtime (crypto.randomUUID()) — evita o erro
  "cannot add postgres_changes after subscribe()" do Strict Mode em dev
- CPF check em tempo real (debounce 500ms) bloqueia botão antes do submit
- Service role isolada em src/lib/supabase/admin.ts — só server actions usam
